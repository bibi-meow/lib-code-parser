<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| caller bytes → libclang in-process parse | Untrusted C++ source crosses into libclang here. libclang parses statically; it never compiles, links, or executes the source. Parse is from in-memory `unsaved_files` (no attacker content written to a predictable temp path). |
| caller `Config.set_library_file` → bundled-ABI integrity | A caller could attempt to swap the libclang shared lib, defeating the DET-02 pin. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-04 | Tampering | bundled libclang ABI vs `Config.set_library_file` | mitigate | Guard rejects any non-None `Config.library_file` and asserts `Config.library_path` resolves into the bundled `clang/native/` dir (LNG-03/DET-02). |
| T-04-05 | Integrity / DoS | libclang FFI memory safety | mitigate | Version assertion uses `importlib.metadata.version` only; the code never touches `conf.lib.*.restype` (Pitfall 2 — segfault reproduced twice in research). Stays on the high-level binding. |
| T-04-06 | DoS | malicious/huge C++ input → libclang OOM/hang | accept | In-process static parse, no codegen; caller controls input size (documented expectation). No subprocess timeout applies. Threat surface narrow for an offline parser. |
| T-04-07 | Info disclosure | `#include` of attacker-controlled paths | accept | `-I` dirs are caller-supplied via `compile_args` (LNG-05); the library never auto-discovers `compile_commands.json` (out of scope). Unresolved includes → warning, not a broader file-read. |
</threat_model>
